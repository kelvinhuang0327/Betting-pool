## 1. Executive Summary

Phase 6N confirmed `HISTORICAL_RECOVERY_NOT_POSSIBLE` after scanning 180 evidence
sources (293 candidate timestamp events, 0 pre-match inference events). The 2,986
existing ML-only model output rows — all generated by the Phase 6L retrospective
adapter from the `mlb_decision_quality_report.json` paper-tracking report — cannot
be assigned a safe `prediction_time_utc` because no pre-match inference ever occurred
and no log evidence survives to map a timestamp to an individual
`canonical_match_id`.

**This document defines the forward-looking architecture** that future model inference
pipelines must implement so that every emitted row carries a natively captured
`prediction_time_utc` and `feature_cutoff_time_utc` at the moment of inference —
enabling M6 `TIMING_VALID` to pass and rows to become CLV candidates.

Key outcomes defined here:

1. A seven-stage inference pipeline with explicit timestamp emission at each stage.
2. Six required timestamp fields extending the Phase 6J contract.
3. Precise semantic definitions for `prediction_time_utc` and `feature_cutoff_time_utc`.
4. Hard-fail leakage guard rules.
5. Validator extension plan (Phase 6P) and a complete implementation roadmap through
   Phase 6U.
6. A ready-to-copy Phase 6P prompt.

**No model code is modified. No predictions are generated. No existing data files are
modified. No DB or runtime changes. No commit. Scope: documentation only.**

## 3. Current State Recap

### 3.1 Why M6 Fails Universally

M6 `TIMING_VALID` requires three conditions (from Phase 6J §7):

| Rule | Condition | Current State |
|---|---|---|
| T1 | `prediction_time_utc < match_time_utc` | `prediction_time_utc = null` on all 2,986 rows — **T1 cannot evaluate** |
| T2 | `feature_cutoff_time_utc <= prediction_time_utc` | `feature_cutoff_time_utc = null` — **T2 cannot evaluate** |
| T3 | training window end `<` `prediction_time_utc` | `prediction_time_utc = null` — **T3 cannot evaluate** |

**Result**: 0/2,986 rows pass M6. 0/2,986 rows have `clv_usable = true`.

### 3.2 Root Cause

The 2,986 rows are a retrospective paper-tracking reconstruction of the 2025 MLB
season. The source (`mlb_decision_quality_report.json`) was generated 2026-04-24,
approximately 365 days after the earliest game (2025-04-24) and 208 days after the
last game (2025-09-28). No model inference ran before any of these matches. No
pre-match timestamp exists in any log, report, or file metadata that could safely
supply `prediction_time_utc`.

### 3.3 Why Historical Recovery Was Rejected

Phase 6N confirmed:

- **Option A (report `generated_at`)**: `2026-04-24T08:27:30Z` is POST_MATCH for all
  2025 games. Also a single batch timestamp — cannot be mapped per `canonical_match_id`.
- **Option C (scheduler log)**: Zero model inference events found in
  `logs/daemon_heartbeat.jsonl` or `logs/runtime_events.jsonl` for the 2025 MLB
  dataset. Logs contain only odds capture and snapshot monitoring events.
- **Option D (fixed pre-match offset)**: Permanently prohibited. Fabricated
  timestamps undermine CLV measurement integrity unconditionally.
- **File mtime**: `FILE_METADATA_LOW_CONFIDENCE` — not auditable, not attributable
  to model inference time.

### 3.4 Permanent Disposition of Existing Rows

The 2,986 rows in `data/derived/model_outputs_2026-04-29.jsonl` must:

- Keep `prediction_time_utc = null`
- Keep `feature_cutoff_time_utc = null`
- Keep `clv_usable = false`
- Carry quality flags: `PREDICTION_TIME_MISSING`, `CANONICAL_MATCH_ID_SYNTHESIZED`,
  `ODDS_SNAPSHOT_REF_MISSING`, `MATCH_TIME_UTC_APPROXIMATE`
- Be treated as historical research-only records, not CLV candidates

No backfill. No timestamp recovery. This is the correct and permanent disposition.

### 3.5 Forward Path

Future inference pipeline components must record `prediction_time_utc` **natively at
inference time** — at the moment the model probability estimate becomes final for a
given row. This Phase 6O document defines the architecture for that capture.

## 4. Native Timestamp Capture Architecture

The future inference pipeline has seven stages. Each stage must emit or reference
at least one timestamp field that is recorded in the model output row.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    FUTURE MODEL INFERENCE PIPELINE                          │
│                                                                             │
│  Stage 1          Stage 2          Stage 3          Stage 4                │
│  ──────────       ──────────       ──────────       ──────────              │
│  Match            Odds Snapshot    Feature          Inference               │
│  Discovery        Availability     Assembly         Start                   │
│                                                                             │
│  ↓ records:       ↓ records:       ↓ records:       ↓ records:              │
│  match_time_utc   odds_snapshot_   feature_cutoff_  prediction_run_         │
│  canonical_       time_utc         time_utc         started_at_utc          │
│  match_id         odds_snapshot_   (max timestamp                           │
│                   ref              of all inputs)                           │
│                                                                             │
│  Stage 5          Stage 6          Stage 7                                 │
│  ──────────       ──────────       ──────────                               │
│  Inference        Row Emission     Validator                                │
│  Complete         (write JSONL)    M6 Check                                │
│                                                                             │
│  ↓ records:       ↓ records:       ↓ checks:                               │
│  prediction_run_  model_output_    prediction_time_utc < match_time_utc    │
│  completed_at_    written_at_utc   feature_cutoff_time_utc <=              │
│  utc              (= prediction_   prediction_time_utc                     │
│  prediction_time_ time_utc for     odds_snapshot_time_utc <=               │
│  utc              synchronous      prediction_time_utc                     │
│                   pipelines)                                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Stage 1 — Match Discovery

| Property | Definition |
|---|---|
| **Timestamp to record** | None emitted at this stage; uses `match_time_utc` from schedule |
| **Source of timestamp** | Official league schedule API or verified data file |
| **Precision expectation** | Minute-level; `YYYY-MM-DDTHH:MM:00Z` |
| **Required field** | `match_time_utc` (already in Phase 6J contract; must be non-null) |
| **Failure handling** | If `match_time_utc` is null or unparseable, abort row emission; emit `MATCH_TIME_MISSING` quality flag |

**Rule**: All downstream stages depend on a valid, parseable `match_time_utc`. Discovery
must occur before inference start. If the scheduled match time is approximate or
unverified, emit `MATCH_TIME_UTC_APPROXIMATE` flag and downgrade row to non-CLV-usable.

### Stage 2 — Odds Snapshot Availability

| Property | Definition |
|---|---|
| **Timestamp to record** | `odds_snapshot_time_utc` — when the odds snapshot used for EV computation was captured |
| **Source of timestamp** | `snapshot_time_utc` field from `odds_snapshots_YYYY-MM-DD.jsonl`; copied from the referenced snapshot record |
| **Precision expectation** | Second-level or sub-minute; `YYYY-MM-DDTHH:MM:SSZ` |
| **Required field** | `odds_snapshot_ref` (Phase 6J contract); `odds_snapshot_time_utc` (new — Phase 6O addendum) |
| **Failure handling** | If no odds snapshot is available before match start: set `odds_snapshot_ref = null`, `odds_snapshot_time_utc = null`, emit `ODDS_SNAPSHOT_REF_MISSING` flag; row cannot compute `expected_value`; row is `clv_usable = false` |

**Rule**: `odds_snapshot_time_utc` must be before `match_time_utc`. If the captured
snapshot post-dates match start, emit `ODDS_SNAPSHOT_AFTER_MATCH` flag and mark row
non-CLV-usable.

### Stage 3 — Feature Assembly

| Property | Definition |
|---|---|
| **Timestamp to record** | `feature_cutoff_time_utc` — the latest timestamp among all input data sources consumed by the model |
| **Source of timestamp** | Max of: (a) `odds_snapshot_time_utc` if odds-derived features are used; (b) player stats dataset cutoff; (c) historical results dataset cutoff; (d) any external data source timestamp |
| **Precision expectation** | Same or finer than odds snapshot; second-level |
| **Required field** | `feature_cutoff_time_utc` (Phase 6J contract; must be non-null for CLV usability) |
| **Failure handling** | If no timestamped feature source exists: `feature_cutoff_time_utc = null`; emit `FEATURE_CUTOFF_MISSING` flag; row is `clv_usable = false` |

**Rule**: Feature assembly code must log the max timestamp of all consumed data sources
and pass it to the row emitter as `feature_cutoff_time_utc`. This cannot be inferred
after the fact.

### Stage 4 — Inference Start

| Property | Definition |
|---|---|
| **Timestamp to record** | `prediction_run_started_at_utc` — UTC wall-clock timestamp at the moment model `.predict()` (or equivalent) is first called for this prediction run |
| **Source of timestamp** | `datetime.utcnow()` / `datetime.now(timezone.utc)` captured in the inference entry-point function, before any model call |
| **Precision expectation** | Millisecond or microsecond; ISO8601 with sub-second component |
| **Required field** | `prediction_run_started_at_utc` (new — Phase 6O addendum) |
| **Failure handling** | If timestamp cannot be captured (system clock error), abort row emission; do not emit a row without `prediction_run_started_at_utc` |

**Rule**: `prediction_run_started_at_utc` must be captured in process, not reconstructed
from file metadata or log parsing. It is the earliest bound on `prediction_time_utc`.

### Stage 5 — Inference Complete

| Property | Definition |
|---|---|
| **Timestamp to record** | `prediction_run_completed_at_utc` — UTC wall-clock timestamp immediately after model `.predict()` returns and probability calibration is applied |
| **Source of timestamp** | `datetime.now(timezone.utc)` captured in the inference exit-point function |
| **Precision expectation** | Millisecond or microsecond |
| **Required field** | `prediction_run_completed_at_utc` (new — Phase 6O addendum); `prediction_time_utc` — set equal to `prediction_run_completed_at_utc` for synchronous pipelines |
| **Failure handling** | If `prediction_run_completed_at_utc < prediction_run_started_at_utc`, emit `TIMESTAMP_CLOCK_DRIFT` flag; abort row emission |

**Rule**: `prediction_time_utc` must be set to `prediction_run_completed_at_utc` for
synchronous single-model pipelines. For ensemble pipelines, `prediction_time_utc` is
the latest `prediction_run_completed_at_utc` among all component models.

### Stage 6 — Row Emission (Write JSONL)

| Property | Definition |
|---|---|
| **Timestamp to record** | `model_output_written_at_utc` — UTC wall-clock timestamp at the moment the JSONL row is written to disk |
| **Source of timestamp** | `datetime.now(timezone.utc)` captured by the row emitter function |
| **Precision expectation** | Millisecond; ISO8601 with sub-second component |
| **Required field** | `model_output_written_at_utc` (new — Phase 6O addendum) |
| **Failure handling** | If write fails, do not record a partial row; surface as write error, not as a timestamp gap |

**Rule**: `model_output_written_at_utc >= prediction_time_utc` must hold. This provides
an upper bound audit trail for each row.

### Stage 7 — Validator M6 Check

| Property | Definition |
|---|---|
| **Timestamp to record** | None — this stage reads, does not write |
| **Source of timestamp** | Reads all six timestamp fields from emitted row |
| **Precision expectation** | N/A |
| **Required field** | All six timestamp fields must be present and parseable for M6 to pass |
| **Failure handling** | If any required timestamp is null or unparseable: M6 fails; `clv_usable = false`; emit `TIMING_VALID_FAIL` quality flag |

**Rule**: The validator (Phase 6K, to be extended in Phase 6P) checks the full
timestamp chain: `feature_cutoff_time_utc ≤ prediction_time_utc < match_time_utc`.
It also verifies `odds_snapshot_time_utc ≤ prediction_time_utc` when an odds snapshot
is referenced.

## 5. Required Timestamp Fields

Six new or promoted timestamp fields are required for a fully CLV-usable model output
row. Three already exist in the Phase 6J contract (with null allowed); three are new
additions from this Phase 6O addendum.

| Field | Phase 6J Status | Phase 6O Status | Nullable for CLV? | Notes |
|---|---|---|---|---|
| `prediction_time_utc` | Required (nullable) | Required, **non-null** | ❌ Must be non-null | Central timing anchor; set = `prediction_run_completed_at_utc` |
| `feature_cutoff_time_utc` | Required (nullable) | Required, **non-null** | ❌ Must be non-null | Max timestamp of all feature inputs |
| `match_time_utc` | Required (non-null) | Required, non-null | ❌ Must be non-null | Scheduled first pitch in UTC |
| `prediction_run_started_at_utc` | **ABSENT** | **NEW — Required** | ❌ Must be non-null | Captured at inference entry point |
| `prediction_run_completed_at_utc` | **ABSENT** | **NEW — Required** | ❌ Must be non-null | Captured at inference exit point |
| `model_output_written_at_utc` | **ABSENT** | **NEW — Required** | ❌ Must be non-null | Captured at JSONL write time |
| `odds_snapshot_time_utc` | **ABSENT** | **NEW — Conditional** | ✅ Null if no odds snapshot | Required when `odds_snapshot_ref` is non-null |

### Timestamp Chain Invariants

For a row to be `clv_usable = true`, all six invariants must hold:

```
[1] feature_cutoff_time_utc   ≤  prediction_time_utc
[2] prediction_time_utc       ≤  prediction_run_completed_at_utc
[3] prediction_time_utc        <  match_time_utc
[4] prediction_run_started_at  ≤  prediction_run_completed_at_utc
[5] prediction_run_completed_at ≤ model_output_written_at_utc
[6] odds_snapshot_time_utc    ≤  prediction_time_utc   (when odds_snapshot_ref non-null)
```

Violation of any invariant causes immediate M6 failure and `clv_usable = false`.

## 6. Prediction Time Semantics

### 6.1 Definition

```
prediction_time_utc = the UTC wall-clock time at which the model's
probability estimate for a given (canonical_match_id, market_key,
selection_key) becomes final and is committed to the row.
```

For a synchronous single-model inference:

```python
# Correct pattern — capture at inference completion
with model_inference_context(match_id, market_key) as ctx:
    raw_prob = model.predict(feature_vector)
    calibrated_prob = platt_scale(raw_prob, a=A, b=B)
    prediction_time_utc = datetime.now(timezone.utc)   # ← capture HERE
    ctx.emit(prediction_time_utc=prediction_time_utc,
             predicted_probability=calibrated_prob, ...)
```

For an ensemble model:

```python
# prediction_time_utc = max of all component completion times
prediction_time_utc = max(
    model_a_completed_at,
    model_b_completed_at,
    calibration_completed_at,
)
```

### 6.2 What `prediction_time_utc` Is NOT

| Incorrect source | Why it is disqualified |
|---|---|
| `report.generated_at` | Batch analysis time; 365+ days post-match for historical rows |
| File `mtime` | Reflects last write, not inference completion |
| Post-processing batch time | Calibration or aggregation after original inference |
| Validation timestamp | Validator run time is a processing artifact |
| Closing odds time | Reflects market movement, not model execution |
| Settlement time | Post-match; leakage by definition |
| Manual entry / estimated | Fabricated; prohibited unconditionally |

### 6.3 Precision and Format

- **Format**: ISO8601 UTC with sub-second precision — `YYYY-MM-DDTHH:MM:SS.ffffffZ`
- **Clock source**: System wall clock (`datetime.now(timezone.utc)`) at inference
  completion. Do not use file timestamps or log-derived approximations.
- **Monotonicity**: `prediction_run_started_at_utc` ≤ `prediction_time_utc` ≤
  `prediction_run_completed_at_utc` ≤ `model_output_written_at_utc`. Any violation
  is a hard failure.

### 6.4 Multi-Market Run Semantics

When a single inference run produces ML, RL, and OU rows for the same match, each
market row has its own `prediction_time_utc` (capturing the completion time for that
market's probability estimate). `prediction_run_started_at_utc` and
`prediction_run_completed_at_utc` are shared across all rows in the same run (they
describe the run as a whole); `prediction_time_utc` is per-row.

## 7. Feature Cutoff Semantics

### 7.1 Definition

```
feature_cutoff_time_utc = the latest timestamp among all data sources
consumed by the model during feature construction for a given
(canonical_match_id, market_key) prediction.
```

It is the **knowledge horizon** of the model at inference time. Any data with a
timestamp after `feature_cutoff_time_utc` must not have been observable by the model.

### 7.2 Computation Rules

```python
# Required: compute feature_cutoff_time_utc as max of all data source timestamps
feature_cutoff_time_utc = max(
    stats_dataset_cutoff_utc,           # player/team stats refresh cutoff
    odds_snapshot_time_utc,             # if odds features used
    historical_results_cutoff_utc,      # if prior game results used as features
    # ... any additional timestamped source
)
```

All data sources must expose a machine-readable cutoff timestamp. If any source has
no machine-readable cutoff:

- Emit `FEATURE_CUTOFF_SOURCE_UNKNOWN` quality flag
- Set `feature_cutoff_time_utc = null`
- Row is automatically `clv_usable = false`

### 7.3 Hard Rules

| Rule | Condition | Violation Action |
|---|---|---|
| FC-1 | `feature_cutoff_time_utc ≤ prediction_time_utc` | Emit `FEATURE_CUTOFF_AFTER_PREDICTION`; `clv_usable = false` |
| FC-2 | `feature_cutoff_time_utc < match_time_utc` | Emit `FEATURE_CUTOFF_AFTER_MATCH`; `clv_usable = false`; potential leakage |
| FC-3 | If odds-derived features used: `feature_cutoff_time_utc ≥ odds_snapshot_time_utc` | If violated: odds snapshot timestamp is inconsistent with feature set; emit `ODDS_CUTOFF_MISMATCH` |
| FC-4 | If no timestamped feature source exists: `feature_cutoff_time_utc = null` | Emit `FEATURE_CUTOFF_MISSING`; `clv_usable = false` |

### 7.4 Relation to Odds Snapshot

When the model uses odds-derived features (e.g., opening line, current market
probability), the odds snapshot referenced by `odds_snapshot_ref` sets a lower bound
on `feature_cutoff_time_utc`:

```
feature_cutoff_time_utc  ≥  odds_snapshot_time_utc
```

This ensures the odds snapshot is accounted for in the knowledge horizon. The full
chain then becomes:

```
odds_snapshot_time_utc  ≤  feature_cutoff_time_utc  ≤  prediction_time_utc  <  match_time_utc
```

## 8. Metadata Contract Addendum

This Phase 6O addendum defines seven additional fields to be added to future model
output rows. These fields extend but do not replace the Phase 6J `"6j-1.0"` contract.
Future rows implementing this addendum use `schema_version: "6o-1.0"`.

### 8.1 New Required Fields

| Field | Type | Nullable | Description |
|---|---|:---:|---|
| `prediction_run_started_at_utc` | ISO8601 | NO | UTC wall-clock at inference entry point (before model `.predict()`) |
| `prediction_run_completed_at_utc` | ISO8601 | NO | UTC wall-clock after model output and calibration complete |
| `model_output_written_at_utc` | ISO8601 | NO | UTC wall-clock at JSONL row write time |
| `prediction_time_source` | string enum | NO | Source classification for `prediction_time_utc` |
| `feature_cutoff_source` | string enum | NO | Source classification for `feature_cutoff_time_utc` |
| `timestamp_capture_version` | string | NO | Semver of the timestamp capture module (e.g. `"timestamp_capture_v1.0.0"`) |
| `odds_snapshot_time_utc` | ISO8601 | YES | UTC time of odds snapshot; null if `odds_snapshot_ref` is null |
| `timestamp_quality_flags` | list[string] | NO | Set of quality flags for timestamp fields; empty list if all timestamps are valid |

### 8.2 Allowed `prediction_time_source` Values

| Value | Meaning | CLV Eligible? |
|---|---|---|
| `MODEL_INFERENCE_RUNTIME` | `prediction_time_utc` captured in-process at model inference completion | ✅ Yes |
| `MODEL_OUTPUT_EMISSION_RUNTIME` | `prediction_time_utc` captured at row write time (for synchronous pipelines where inference + emission are atomic) | ✅ Yes |
| `SCHEDULER_RUN_RUNTIME` | `prediction_time_utc` derived from a scheduler job event that triggered inference and whose start time is recorded in an inference-specific log | ✅ Yes, with caveats — only if the scheduler event maps 1:1 to the inference run |
| `REPORT_METADATA` | Derived from a batch report `generated_at` field | ❌ No — rejected (Phase 6M §5 Option A) |
| `FILE_METADATA_LOW_CONFIDENCE` | Derived from file `mtime` | ❌ No — rejected (Phase 6N §6) |
| `UNKNOWN` | Source cannot be determined | ❌ No |

### 8.3 Allowed `feature_cutoff_source` Values

| Value | Meaning | CLV Eligible? |
|---|---|---|
| `NATIVE_DATA_SOURCE_CUTOFF` | Cutoff timestamp read from the data source's own metadata field | ✅ Yes |
| `ODDS_SNAPSHOT_TIME` | Cutoff set to odds snapshot time (when odds are the latest feature input) | ✅ Yes |
| `STATS_REFRESH_CUTOFF` | Cutoff from a verifiable stats dataset refresh timestamp | ✅ Yes |
| `INFERRED_FROM_FEATURE_DATE` | Cutoff inferred from the date portion of feature data (no sub-day precision) | ⚠️ Conditional — if precision is sufficient for the match schedule |
| `UNKNOWN` | Source cannot be determined | ❌ No |

### 8.4 `timestamp_quality_flags` Vocabulary

| Flag | Meaning | Triggers CLV Block? |
|---|---|---|
| `TIMESTAMP_CAPTURED_NATIVE` | All required timestamps captured in-process at inference time | No — positive indicator |
| `FEATURE_CUTOFF_NATIVE` | `feature_cutoff_time_utc` computed from machine-readable data source cutoffs | No — positive indicator |
| `ODDS_SNAPSHOT_REF_NATIVE` | `odds_snapshot_ref` and `odds_snapshot_time_utc` from verifiable snapshot | No — positive indicator |
| `PREDICTION_TIME_AFTER_MATCH` | `prediction_time_utc ≥ match_time_utc` | ✅ Yes — hard fail |
| `FEATURE_CUTOFF_AFTER_PREDICTION` | `feature_cutoff_time_utc > prediction_time_utc` | ✅ Yes — hard fail |
| `TIMESTAMP_SOURCE_LOW_CONFIDENCE` | Any timestamp derived from file metadata or batch report `generated_at` | ✅ Yes |
| `TIMESTAMP_MISSING` | Any required timestamp field is null | ✅ Yes |
| `TIMESTAMP_CLOCK_DRIFT` | `prediction_run_started_at_utc > prediction_run_completed_at_utc` | ✅ Yes |
| `ODDS_CUTOFF_MISMATCH` | `feature_cutoff_time_utc < odds_snapshot_time_utc` | ✅ Yes |
| `ODDS_SNAPSHOT_AFTER_MATCH` | `odds_snapshot_time_utc ≥ match_time_utc` | ✅ Yes |
| `HISTORICAL_TIMESTAMP_RECOVERY` | Timestamp was recovered from historical evidence (not native capture) | ✅ Yes |

### 8.5 Schema Version Progression

| Version | Scope | Key Addition |
|---|---|---|
| `"6j-1.0"` | Phase 6J contract | 31 required fields; `prediction_time_utc` nullable |
| `"6o-1.0"` | Phase 6O addendum | +7 fields; `prediction_time_utc` non-null required; timestamp chain invariants |

All existing `"6j-1.0"` rows from Phase 6L retain their schema version. They are not
upgraded or modified. New inference pipeline rows emit `"6o-1.0"`.

## 9. Leakage Guard Rules

The following conditions constitute hard-fail leakage violations. Any row triggering a
hard-fail rule is automatically `clv_usable = false` and must not enter CLV computation.

| Rule ID | Condition | Quality Flag | Action |
|---|---|---|---|
| LG-01 | `prediction_time_utc ≥ match_time_utc` | `PREDICTION_TIME_AFTER_MATCH` | Hard fail — leakage by definition |
| LG-02 | `feature_cutoff_time_utc > prediction_time_utc` | `FEATURE_CUTOFF_AFTER_PREDICTION` | Hard fail — feature knowledge exceeds prediction moment |
| LG-03 | `odds_snapshot_time_utc > prediction_time_utc` | `ODDS_SNAPSHOT_AFTER_MATCH` (or `ODDS_CUTOFF_MISMATCH`) | Hard fail — odds snapshot post-dates prediction |
| LG-04 | Any settlement or result field present with a value set before `prediction_time_utc` | `FUTURE_LEAKAGE` | Hard fail — actual result used as feature |
| LG-05 | `prediction_time_source` is `REPORT_METADATA`, `FILE_METADATA_LOW_CONFIDENCE`, or `UNKNOWN` | `TIMESTAMP_SOURCE_LOW_CONFIDENCE` | Hard fail for CLV |
| LG-06 | `TIMESTAMP_MISSING` in `timestamp_quality_flags` | `TIMESTAMP_MISSING` | Hard fail — cannot evaluate timing |
| LG-07 | `TIMESTAMP_CLOCK_DRIFT` in `timestamp_quality_flags` | `TIMESTAMP_CLOCK_DRIFT` | Hard fail — clock inconsistency |
| LG-08 | `HISTORICAL_TIMESTAMP_RECOVERY` in `timestamp_quality_flags` | `HISTORICAL_TIMESTAMP_RECOVERY` | Hard fail — recovered timestamps not CLV-safe |
| LG-09 | `feature_cutoff_time_utc ≥ match_time_utc` | `FEATURE_CUTOFF_AFTER_MATCH` | Hard fail — feature uses post-start data |
| LG-10 | Training window end ≥ `prediction_time_utc` (T3 from Phase 6J) | `TRAINING_WINDOW_OVERLAP` | Hard fail — training data overlaps prediction time |

### 9.1 Soft Warnings (Non-Blocking)

| Rule ID | Condition | Quality Flag | Action |
|---|---|---|---|
| LW-01 | `prediction_time_utc` captured > 24 hours before `match_time_utc` | `PREDICTION_VERY_EARLY` | Warning only — logged; row may still be CLV candidate |
| LW-02 | `model_output_written_at_utc > prediction_time_utc + 5 minutes` | `EMIT_LATENCY_HIGH` | Warning only — pipeline latency tracked |
| LW-03 | `odds_snapshot_time_utc` > 2 hours before `match_time_utc` | `ODDS_SNAPSHOT_STALE` | Warning only — odds may not reflect pre-game market |

## 10. Validator Impact

The Phase 6K validator (`scripts/validate_model_output_contract.py`) currently
implements M1–M12 gates against the Phase 6J `"6j-1.0"` schema. Phase 6O requires
the following extensions, to be implemented in Phase 6P:

### 10.1 M6 Extension (`M6_TIMING_VALID`)

**Current M6 checks** (Phase 6K):
- T1: `prediction_time_utc < match_time_utc`
- T2: `feature_cutoff_time_utc ≤ prediction_time_utc`
- T3: training window end < `prediction_time_utc`

**Extended M6 checks** (Phase 6P — do not implement now):

```
T4: prediction_run_started_at_utc ≤ prediction_time_utc
T5: prediction_time_utc ≤ prediction_run_completed_at_utc
T6: prediction_run_completed_at_utc ≤ model_output_written_at_utc
T7: odds_snapshot_time_utc ≤ prediction_time_utc  (when odds_snapshot_ref non-null)
```

### 10.2 M9 Extension (`M9_NO_LEAKAGE_HARD_FAIL`)

**Current M9 checks** (Phase 6K):
- Checks for settlement leakage fields (`actual_result`, `home_score`, etc.)
- Checks for `FUTURE_LEAKAGE` / `POST_MATCH` / `RESULT_USED` flags

**Extended M9 checks** (Phase 6P — do not implement now):

```
Check prediction_time_source is not in disallowed set
Check timestamp_quality_flags does not contain any hard-fail flags
  (TIMESTAMP_MISSING, TIMESTAMP_SOURCE_LOW_CONFIDENCE, HISTORICAL_TIMESTAMP_RECOVERY,
   TIMESTAMP_CLOCK_DRIFT, FEATURE_CUTOFF_AFTER_MATCH)
```

### 10.3 M11 Extension (`M11_CLV_USABLE_FLAG_CORRECT`)

**Current M11 checks** (Phase 6K):
- `clv_usable = false` if prediction_time_utc is null, or if any hard-fail quality flag present

**Extended M11 checks** (Phase 6P — do not implement now):

```
clv_usable = true requires:
  - prediction_time_source in {MODEL_INFERENCE_RUNTIME,
      MODEL_OUTPUT_EMISSION_RUNTIME, SCHEDULER_RUN_RUNTIME}
  - timestamp_quality_flags contains TIMESTAMP_CAPTURED_NATIVE
  - schema_version == "6o-1.0" (or later)
  - all six timestamp chain invariants satisfied (§5)
  - no hard-fail leakage guard rules triggered (§9)
```

### 10.4 New Gate M13 (Phase 6P)

A new gate `M13_TIMESTAMP_CHAIN_VALID` should be added:

```
M13_TIMESTAMP_CHAIN_VALID — passes only when all six timestamp chain
invariants (§5) hold and prediction_time_source is an allowed CLV source.
M13 failure → clv_usable = false; emit TIMESTAMP_CHAIN_INVALID flag.
```

**Do not implement M13 now.** It is specified here for Phase 6P.

- Must use `schema_version: "6o-1.0"`

3. Have a corresponding closing odds snapshot in the odds pipeline
4. Pass all M1–M13 gates

Historical Phase 6L rows are permanently excluded from CLV measurement. They may be
used for paper-tracking analysis, model diagnostics, and backtest research only.
